Active Directory Attacks

Understanding Common AD Attack Vectors & Mitigations

Active Directory (AD) is a critical component of most Windows network environments, but it's also a primary target for attackers. Understanding these attack techniques is essential for both penetration testers and defenders.

How Kerberos Authentication Works in AD:

  • When a user logs onto their workstation, their machine will send an AS-REQ message to the Key Distribution Center (KDC), aka Domain Controller, requesting a TGT using a secret key derived from the userâ€™s password.
  • The KDC verifies the secret key with the password it has stored in Active Directory for that user. Once validated, it returns the TGT in an AS-REP message. The TGT contains the user's identity and is encrypted with the KDC secret key (the krbtgt account).
  • When the user attempts to access a resource backed by Kerberos authentication (e.g. a file share), their machine looks up the associated Service Principal Name (SPN). It then requests (TGS-REQ) a Ticket Granting Service Ticket (TGS) for that service from the KDC, and presents its TGT as a means of proving they're a valid user.
  • The KDC returns a TGS (TGS-REP) for the service in question to the user, which is then presented to the actual service. The service inspects the TGS and decides whether it should grant the user access or not.

    Kerberos & Active Directory Concepts

    Understanding these fundamental concepts is essential to grasp how AD attacks work:

    TGT - Ticket Granting Ticket

    A TGT is issued by the Authentication Service (AS) during the initial authentication. It allows the user to request service tickets (TGS) without re-entering credentials for a certain period (usually 10 hours).

    TGS - Ticket Granting Service

    A service ticket issued by the Ticket Granting Service to allow a user to access a specific network service. The TGS is encrypted with the service account's password hash.

    KRBTGT Account

    A special domain account used to encrypt and sign all TGTs in the domain. Compromising the KRBTGT account hash allows attackers to create Golden Tickets.

    SPN - Service Principal Name

    A unique identifier for a service instance, used to associate a service with a domain account. Format: service_class/host:port/service_name. Required for Kerberoasting attacks.

    SID - Security Identifier

    A unique value that identifies a user, group, or computer account in Windows. Domain SID format: S-1-5-21-3623811015-3361044348-30300820-

    NT Hash (NTLM Hash)

    The result of hashing a password with the MD4 algorithm, used for NTLM authentication. Stored in the SAM database or NTDS.dit. Example: aad3b435b51404eeaad3b435b51404ee (empty password hash).

    Kerberos Authentication Flow

    1. AS-REQ: User requests TGT
    2. AS-REP: DC returns encrypted TGT
    3. TGS-REQ: User requests service ticket
    4. TGS-REP: DC returns service ticket
    5. AP-REQ: User presents ticket to service

    When a user logs in:
    1. User enters credentials
    2. LSASS:
    - Derives a Kerberos key from the password (or uses cached creds)
    - Sends an AS-REQ to the KDC (DC)
    3. KDC responds with AS-REP containing a TGT
    4. LSASS stores the TGT in memory
    ðŸ‘‰ The TGT is not stored on disk, it lives in LSASS memory.

    NTDS.dit

    The AD database file containing all domain objects, including password hashes. Located on domain controllers at %SystemRoot%\NTDS\NTDS.dit. Often targeted in credential dumping attacks.

    LSASS

    lsass.exe a core Windows security process handles authentication/authorization, for example: verify logged in users activity, handle passwords changes, create access tokens. it holds all the hashes, passwords and tokens across windows domain in memory for the loggedin users.


    LSASS process part of os and runs as root/authority system


    Also support in Kerberos tickets granting, When a user logs in, LSASS is the process that actually talks Kerberos to the Domain Controller (KDC).

    SAM - The Security Account Manager

    (SAM) database holds the NTLM hashes of local accounts only

    Delegation

    Delegation allows a user or machine to act on behalf of another user to another service. A common implementation of this is where a user authenticates to a front-end web application that serves a back-end database. The front-end application needs to authenticate to the back-end database (using Kerberos) as the authenticated user.

    Key Insight: Most AD attacks exploit weaknesses in how Kerberos tickets are issued, validated, or delegated. Understanding the normal flow helps identify attack vectors.

    Active Directory Attacks

    These attacks typically require initial access to the network, often through phishing, exploitation of vulnerabilities, or other means.

    Golden Ticket Attack

    Description

    Forged Kerberos Ticket Granting Ticket (TGT) that provides unrestricted access to any resource in the domain. Created using the KRBTGT account's hash.

    Requirements

    Domain admin privileges or KRBTGT account hash, Domain SID, and knowledge of the domain name.

    Tools
    Mimikatz Impacket Rubeus
    Impact

    Critical: Complete domain compromise, persistence, and bypass of authentication.

    Example Commands
    Mimikatz:
    kerberos::golden /admin:Administrator /domain:domain.local /sid:S-1-5-21-... /krbtgt:a9b30e5f... /ptt
    Rubeus:
    Rubeus.exe golden /krbtgt:a9b30e5f... /domain:domain.local /sid:S-1-5-21-... /user:Administrator /ptt
    OPSEC Considerations
    Detection Risks:
    â€¢ DC event 4769 (Kerberos TGT requested) with abnormal encryption type (0x17 for RC4)
    â€¢ DC event 4672 (Special privileges assigned) when using Golden Ticket
    â€¢ Tickets with extremely long lifetimes (default: 10 years)
    Stealth Tips:
    â€¢ Use Diamond Ticket instead for better stealth
    â€¢ Set reasonable ticket lifetime (not 10 years)
    â€¢ Use from a compromised host rather than attacker machine
    â€¢ Reset KRBTGT password twice to invalidate all Golden Tickets

    Silver Ticket Attack

    Description

    Forged Kerberos service ticket (TGS) for a specific service. Doesn't require communication with DC as it's signed with service account hash.

    Requirements

    Service account hash, target service SPN, domain SID, and target username.

    Tools
    Mimikatz Impacket Rubeus
    Impact

    High: Access to specific service, lateral movement, and persistence.

    Example Commands
    Mimikatz:
    kerberos::golden /user:Administrator /domain:domain.local /sid:S-1-5-21-... /target:server.domain.local /service:HOST /rc4:service_hash /ptt
    Creating CIFS ticket:
    kerberos::golden /domain:domain.local /sid:SID /user:user /service:cifs /target:server.domain.local /rc4:hash /ptt
    OPSEC Considerations
    Detection Risks:
    â€¢ Service logs will show access but no corresponding TGT request on DC
    â€¢ Event 4769 on DC missing for the Silver Ticket usage
    â€¢ PAC validation failures if ticket is poorly forged
    Stealth Tips:
    â€¢ Use legitimate user SIDs that exist in the domain
    â€¢ Set appropriate ticket timestamps (not future dates)
    â€¢ Limit use to targeted services only
    â€¢ Use from already compromised systems

    Diamond Ticket Attack

    Description

    Modified legitimate TGT ticket rather than forged from scratch. More stealthy than Golden Ticket as it uses real encrypted timestamps.

    Requirements

    KRBTGT hash, legitimate TGT ticket, ability to decrypt and re-encrypt ticket.

    Tools
    Rubeus Custom tools
    Impact

    Critical: Domain persistence with better stealth than Golden Ticket.

    Kerberoasting

    Description

    Requesting service tickets for accounts with SPNs and then cracking them offline to obtain plaintext passwords.

    Requirements

    Domain account (any), service accounts with SPNs, and weak service account passwords.

    Tools
    Rubeus Impacket Hashcat John the Ripper
    Impact

    High: Obtain service account credentials, potentially leading to privilege escalation.

    Example Commands
    Rubeus:
    Rubeus.exe kerberoast /outfile:hashes.txt
    Impacket:
    python3 GetUserSPNs.py -request -dc-ip 10.10.10.10 domain.local/user
    PowerShell:
    Add-Type -AssemblyName System.IdentityModel; New-Object System.IdentityModel.Tokens.KerberosRequestorSecurityToken -ArgumentList "MSSQLSvc/sqlserver.domain.local:1433"
    Cracking:
    hashcat -m 13100 hashes.txt rockyou.txt -O
    OPSEC Considerations
    Detection Risks:
    â€¢ DC event 4769 (service ticket requested) for multiple SPNs
    â€¢ Large number of TGS requests from single user in short time
    Stealth Tips:
    â€¢ Request tickets slowly over time (not all at once)
    â€¢ Target specific SPNs rather than all
    â€¢ Use legitimate domain accounts for requests
    â€¢ Crack offline on separate system

    ASREP Roasting

    Description

    Targeting accounts with "Do not require Kerberos pre-authentication" enabled, an AS-REP can be requested for that user throught TGT request, and part of the reply include TGT can be cracked offline to recover plaintext password.

    Requirements

    Accounts with pre-authentication disabled and weak passwords.

    Tools
    Rubeus Impacket Hashcat
    Impact

    Medium: Obtain user credentials if passwords are weak.

    Example Commands

    Rubeus.exe asreproast /format:hashcat /outfile:asrep_hashes.txt
    Impacket:
    python3 GetNPUsers.py domain.local/ -usersfile users.txt -format hashcat -outputfile hashes.txt
    Cracking:
    hashcat -m 18200 asrep_hashes.txt rockyou.txt -O
    PowerView to find users:
    Get-DomainUser -PreauthNotRequired | select samaccountname
    OPSEC Considerations
    Detection Risks:
    â€¢ DC event 4768 with error code 0x0 (PREAUTH_NOT_REQUIRED)
    â€¢ Multiple AS-REQ requests for different users
    â€¢ Unusual source IPs requesting AS-REP tickets

    â€¢ Target specific known vulnerable accounts only
    â€¢ Perform reconnaissance first to identify target accounts
    â€¢ Request during normal business hours
    â€¢ Use legitimate domain workstations when possible
